# 头脑风暴整理

## 一句话定位

GetMyPackage 是一个 Offer 真实年包计算器：把工资、奖金、股票、期权、福利、假期、五险一金、税、通勤、餐补、住房、风险和时间成本全部折现，帮助用户比较不同公司的真实收益。

## 产品关键词

- 年包包
- 真实年包
- Offer 折现
- 福利量化
- 大厂 Offer 对比
- 现金流视角
- 公司福利账本
- 隐私优先

## 图标方向

icon：财神兽年包包。

可以是一个抱着红包/包袱/金币袋的“财神兽”，既有财富感，又不要像严肃金融工具。风格建议：圆润、可信、轻量、略带好运感，避免博彩感。

## 核心用户

1. 拿到多个 Offer 的互联网/科技行业候选人。
2. 正在考虑跳槽、转岗、异地工作的员工。
3. 想知道自己当前公司“真实收入”的在职员工。
4. HR/猎头/职业咨询者，但这类用户后期再考虑。

## 核心场景

- A 公司 base 高，但通勤远、加班多、三餐少，真实年包可能没看起来高。
- B 公司股票多，但解禁周期长、波动大、兑现概率不确定，需要折现。
- C 公司福利多，包括餐补、班车、补充医疗、健身房、带薪假，应该计入收益。
- 异地 Offer 需要考虑房租、通勤、城市税后收入、生活成本、机会成本。
- 用户可以创建多个 Offer，一键对比“名义年包”和“真实折现年包”。

## 计算思路

真实年包 = 税后现金收入 + 折现后的股权/期权 + 可货币化福利 + 时间价值 + 风险调整收益 - 工作相关支出 - 风险成本

其中：

- 税后现金收入：base、固定奖金、绩效奖金、签字费、补贴。
- 股权/期权：按 vesting、行权价、当前估值、流动性、波动率、兑现概率折现。
- 福利：三餐、餐补、班车、补充医疗、商业保险、健身房、员工折扣、年假、病假、学习预算。
- 工作相关支出：通勤费、停车费、租房溢价、搬家成本、着装设备、外食、育儿/家政替代成本。
- 时间价值：通勤时间、加班时间、可远程办公天数、假期天数。
- 风险调整：裁员风险、绩效不确定性、奖金不确定性、股权流动性、试用期风险。

## 默认指标与可调节建议

产品应该给出默认估算，但允许用户调整：

- 折现率：默认 5%-8%，可切换保守/中性/乐观。
- 通胀率：默认 2%-3%，支持用户自定义。
- 股票风险折扣：上市公司较低，未上市公司较高。
- 期权流动性折扣：未上市期权默认大幅折价。
- 餐食价值：按用户真实替代消费估算，而不是公司标价。
- 通勤时间价值：可按用户税后时薪、半时薪或自定义值计算。
- 加班成本：既可以作为负收益，也可以只进入生活质量评分。

## 数据来源

### 1. 用户自定义

最重要，也最安全。用户输入自己的 Offer、城市、通勤、福利、税前税后偏好。

### 2. 本地默认数据

内置一些公开、非敏感、可解释的默认值，例如：

- 城市社保公积金规则
- 个税规则
- 常见通胀/折现率假设
- 常见福利估值模板
- 行业默认风险参数

### 3. 用户共建

用户匿名贡献公司福利信息，但要非常谨慎：

- 不收集个人身份信息
- 不展示小样本敏感数据
- 做区间和聚合
- 标注更新时间、样本数量、可信度

### 4. 爬取公开网页

只爬公开信息，优先用官方/公开来源：招聘页、财报、福利介绍、公开政策、新闻。不要爬需要登录、违反 ToS 或包含个人隐私的信息。

## 程序端/数据端分离

建议结构：

- app：前端 Web App，本地运行。
- core：计算引擎，纯函数，方便测试。
- data：默认参数、城市规则、公司福利模板。
- research：用户调研、竞品分析、访谈记录。
- docs：产品文档、风险文档、版本计划。

早期可以先用 JSON/YAML 保存数据，后面再接数据库。

## 本地 1.0 功能范围

1. 创建 Offer。
2. 输入基础薪资、奖金、补贴、股票/期权。
3. 输入福利：餐、班车、保险、假期、学习预算等。
4. 输入成本：通勤、住房差异、加班、搬家等。
5. 设置折现率、通胀率、风险折扣。
6. 输出：名义年包、税后年包、真实折现年包、福利价值、成本扣减、风险调整。
7. 支持两个或多个 Offer 对比。
8. 所有数据默认存在本地。

## 高压线与风险

### 隐私

Offer、薪资、公司、职级、地点都高度敏感。默认应本地存储，不上传；上传必须明确告知并匿名化。

### 合规

税务、理财、投资建议需要免责声明。不要直接告诉用户“应该买/卖/投什么”，只能做假设测算和教育性说明。

### 公司数据

避免展示可反推个人身份的小样本数据。公司福利数据要有来源、时间和置信度。

### 爬虫

只抓公开网页，遵守 robots.txt 和网站条款，不绕过登录、验证码或权限限制。

## 第三方理财思路

早期不建议深度引入第三方理财。可以先做：

- 折现率解释器
- 现金流模拟器
- RSU/期权风险提示
- 税后收入和储蓄率估算

后期如果引入第三方服务，只做可选插件，并明确区分“计算工具”和“理财建议”。

## 最有价值的产品差异

不是再做一个薪资计算器，而是做“工作总账本”：公司给你的每一瓶水、每一顿饭、每一天假期、每一次通勤、每一个加班小时，最后都进入同一套可解释的折现模型。

## 2026-07-08 产品边界补充

### 1.0 上线原则：用户端优先

1.0 版本应该是一个数据全在用户端的本地/静态 Web App。项目方只提供：

- 计算公式
- 默认假设
- 可解释的输入项
- 漂亮、可信、好用的 UI
- 可选的第三方公开数据入口，例如 GitHub Pages 上的静态数据文件

1.0 不存用户隐私数据，不要求注册账号，不做后端数据库。用户的薪资、Offer、公司、职级、城市、家庭偏好等敏感信息默认只保存在本地。

后续在隐私、合规、用户授权、数据脱敏和安全设计都考虑周全后，再考虑上线后端：

- 注册账户
- 跨设备同步
- 加密存储用户隐私数据
- 用户可控的数据删除和导出
- 匿名聚合数据分析
- 用匿名数据优化 Offer 模型和默认参数

### 第三方数据接入方式

早期只接入低风险、静态、公开的数据源。优先考虑 GitHub Pages / 静态 JSON / 版本化数据文件，而不是直接把用户数据传到服务端。

第三方数据只用于辅助默认参数，例如城市规则、税率规则、福利估算模板、行业假设等。用户自己的 Offer 数据不应被上传。

## 量化分层原则

不是所有东西都应该直接进入计算。产品需要把因素分成两层：

### 进入计算的硬指标

这些因素可以相对明确地折算成钱，应该进入真实年包模型：

- 税前/税后现金
- 奖金
- 补贴
- RSU/期权折现
- 五险一金
- 三餐自付成本变化
- 通勤现金支出
- 房租/搬家/异地成本
- 工作导致的必要开销

### 只提醒、不计入的软指标

这些因素重要，但每个人权重差异太大，早期不应假装精确计算。产品应该提醒用户认真考虑，但默认不进入真实年包数字：

- 离家近不近
- 通勤时间的主观痛苦
- 团队氛围
- 管理风格
- 高压程度
- 成长性
- 简历增益
- 是否喜欢业务方向
- 家庭照顾便利性
- 城市偏好

例如：通勤价格可以计入计算；通勤时间可以展示和提醒，但不默认折算成钱。用户如果很想量化，也可以自己输入一个主观时间成本参数。

## 福利计算方向修正

福利不应该被粗暴当成“加钱”。核心逻辑应该是：

> 我为了上班额外花的钱，都应该从真实年包里扣掉；公司提供的福利，最多是减少这部分扣减。

因此，班车、免费打车、免费停车等通勤福利通常不是加法，而是把通勤现金成本从负数抵消到 0。

示例：

- 没有班车：每天通勤花 30 元，一年工作 240 天，则扣减 7200 元。
- 有免费班车：通勤现金成本变成 0，则扣减 0 元。
- 不应再额外给“班车福利 +7200 元”，否则会双重计算。

同理，免费三餐也要看用户的真实替代成本：如果用户本来会自己带饭或低成本吃饭，不能按公司标价全额加钱。更合理的做法是计算“因工作导致的餐饮自付成本减少多少”。

## 更准确的真实年包公式

真实年包 = 税后现金收入 + 风险调整后的股权/期权 + 可抵扣的工作成本减少额 - 工作导致的必要支出 - 风险折价

其中，福利更多是“成本抵扣项”，不是天然收入项。

## 2026-07-08 目标用户画像补充

1.0 的目标用户先收窄到中国互联网大厂语境，不做泛职业、泛行业的薪酬工具。

核心用户包括：

1. 中国互联网大厂员工：想评估自己现在这份工作的真实年包，判断是否值得继续待、是否该跳槽。
2. 中国互联网大厂准员工：已经拿到或即将拿到 Offer，需要理解 base、奖金、股票、福利、城市、通勤等综合价值。
3. 大学生和应届生：面对校招 Offer、实习转正、不同城市和不同公司的选择，需要一个能解释复杂薪酬结构的工具。
4. 正在抉择多个 Offer 的候选人：需要把多个 Offer 放到同一套口径下比较，而不是只看名义总包。
5. 对当前工作做复盘的人：不一定马上跳槽，但想知道这份工作扣掉通勤、餐饮、住房、加班、压力等之后到底值多少钱。

因此 1.0 的语言、默认字段、示例数据和 UI 都应围绕中国互联网公司常见薪酬结构设计：月薪、年终奖、绩效、签字费、RSU/期权、五险一金、公积金、餐补、班车、打车、补充医疗、年假、大小周/加班、城市差异等。

## 2026-07-08 核心场景修正

产品核心场景应从“比较多个 Offer”修正为“评估单个 Offer 或当前工作”。

用户真正的核心动作是：把一份 Offer 或现在正在上的班摆在眼前，拆开看它的真实价值。多 Offer 对比只是这个能力自然生出来的附属功能：当用户评估了多个对象后，再把结果放在一起比较。

因此 1.0 的主流程应该是：

1. 创建一个待评估对象：新 Offer、当前工作、实习 Offer、转岗机会或异地机会。
2. 输入现金、股权、福利、成本、风险和主观提醒项。
3. 得到这一个对象的真实年包和解释报告。
4. 可选：保存多个评估结果，再进入对比视图。

这会影响产品定位：GetMyPackage 首先是“Offer/工作真实价值评估器”，其次才是“多 Offer 对比器”。
