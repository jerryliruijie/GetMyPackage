# 核心使用场景

## 核心判断

GetMyPackage 1.0 的核心场景不是先比较多个 Offer，而是把一份 Offer 或一份当前工作摆在用户面前，帮用户评估它到底值多少钱。

多 Offer 对比是自然延伸：当用户已经评估了多个 Offer/工作之后，把多个结果并排展示即可。比较功能不应该反过来主导产品结构。

## 主场景：评估一个 Offer 或当前工作

用户进入产品时，最重要的动作是创建一个“待评估对象”：

- 一个新 Offer
- 一份当前工作
- 一个实习/校招 Offer
- 一个转岗机会
- 一个异地机会

产品要回答的问题是：

> 这份机会扣掉税、风险、通勤、住房、餐饮、加班、工作必要支出之后，真实价值是多少？

## 附属场景：比较多个评估结果

比较不是独立产品核心，而是评估完成后的自然动作。

用户可以先评估 A，再评估 B，然后进入对比页看：

- 名义年包差异
- 税后年包差异
- 真实折现年包差异
- 工作成本差异
- 风险差异
- 未计入计算但需要人工判断的提醒项

## 对产品设计的影响

### 首页

首页应该直接引导用户“评估一份 Offer/工作”，而不是先让用户创建一组比较。

### 信息架构

核心对象应该叫 evaluation / package / work package，而不是 comparison。Comparison 只是一个 view。

### 结果页

结果页应该先讲清楚单个对象：

- 它的名义价值
- 它的真实价值
- 哪些项目被扣掉
- 哪些福利只是抵扣成本
- 哪些因素没有进入计算但需要注意

### 对比页

对比页是多个结果卡片的组合，不承担解释所有模型细节的责任。

## 产品表达

更准确的定位可以是：

- Offer 真实价值评估器
- 工作真实价值账本
- 年包包：帮你拆开一份 Offer 的真实含金量

而不是单纯的 Offer 对比器。
