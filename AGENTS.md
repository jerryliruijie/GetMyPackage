# Agent 开发规则

## 分支与合并

- 禁止直接在 `main` 上开发或提交。每项改动都从最新的 `main` 创建目的明确的独立分支。
- 只有相关的本地验证和远端 CI 全部通过后，才允许通过 Pull Request 合并到 `main`。
- 验证失败时留在当前分支修复，不得绕过检查或把未验证代码合入 `main`。

## 提交规范

- 一个可以独立验证、独立回退的小功能或修复就是一个 commit；不要把多个无关闭环塞进超大 commit。
- commit message 必须让人能直接看懂改了什么以及改在哪里，使用 `feat: ...`、`fix: ...`、`docs: ...`、`test: ...`、`refactor: ...`、`chore: ...` 等 Conventional Commits 前缀。
- 只暂存当前闭环涉及的文件，不得顺带提交用户的无关改动。

## 实现原则

- 代码保持简洁，优先使用现有结构和直接方案；避免不必要的依赖、抽象层、配置项和预判式实现。
- 修改范围只覆盖当前需求。发现相邻问题时，除非它会阻断当前闭环，否则记录后单独处理。
- 验证应与风险相匹配：至少运行相关的类型检查、构建或测试；页面改动还要检查桌面端、移动端和关键交互。

## 本地优先与后端演进

- 1.x 阶段继续坚持本地优先：Offer、薪资、公司、职级、城市和个人偏好等敏感数据默认只保存在用户浏览器，不引入隐式上传、账号或云同步。
- 保持计算逻辑、领域类型、持久化和 UI 的边界清晰。UI 不直接读写底层存储，计算逻辑不依赖浏览器或未来后端。
- 持久化数据必须带版本；不兼容的数据结构变化需要迁移策略，不能静默丢弃或错误复用旧数据。
- 设计新功能时要考虑未来可替换为后端或远端数据源，但在需求出现前不提前搭建 API、服务层或账户体系。
